## 1. 如何处理两类数据集

训练集包含两种文档类型，结构完全不同，走**类型检测 → 分流处理**：

| 类型 | 识别特征 | 规则集 | 判断策略 |
|------|---------|--------|---------|
| **计划任务书** | 标题含"科技项目计划任务书" | `rules/计划任务书/R05-R07.yaml` | 正则判结构 + LLM 多维语义评分 |
| **立项申请书** | 标题含"职工技术创新项目立项申请书" | `rules/立项申请书/R01-R04.yaml` | 纯正则结构规则（100% 准确率，零 API 调用） |

类型检测在管线第一步（parse 节点）完成，通过文档前 500 字符的关键词匹配 `detect_doc_type()`。

两份规则集存放在不同子目录，运行时按类型加载，互不干扰。

## 2. 智能体/模块如何分工

采用 **3 节点顺序管线**（非 LangGraph StateGraph——3 节点线性流程不需要状态机）：

```
[parse] ──DocFields──> [match] ──RuleVerdict[]──> [judge] ──> FinalResult
 文档解析              规则匹配                  汇总裁决
```

### 节点 1：文档解析 (parse)

- **输入**：原始文档文本（doc-read CLI 输出的 Markdown）
- **输出**：`DocFields`（标题、摘要、结构化字段、原文全文）
- **双模式**：正则快速提取（`quick_extract`，无 API 调用）和 LLM 深度结构化
- **关键**：`raw_excerpt` 保留全文，供后续规则引擎和 LLM 检查原始文本

### 节点 2：规则匹配 (match)

- **输入**：`DocFields` + intent + 规则库
- **输出**：`RuleVerdict[]`（每条规则的评估结果、证据、置信度）
- **核心创新**：**正则判结构，LLM 判语义**
  - 正则快速检查：模板残留（R-05）、占位符成员（R-06）、审批在场性（R-01/R-02）、KPI 自我重复（R-07）
  - LLM 语义判断：内容质量评估，R-07 走三维独立评分（技术具体性/KPI 自洽性/预算合理性）
- **Intent 路由**：按意图关键词激活规则子集，不全量暴露

### 节点 3：综合裁决 (judge)

- **输入**：所有 RuleVerdict
- **输出**：`FinalResult`（硬标签 + 命中规则 + 判断依据）
- **裁决铁律**（确定性逻辑）：
  1. 所有规则通过 → "通过"
  2. 任一 critical 规则（confidence ≥ 0.7）不通过 → "不通过"
  3. 模糊情况 → LLM 综合裁决（倾向从严）

### 数据契约

Agent 间通过 Pydantic 模型交接（`schemas.py`）——松耦合，每个节点可独立测试。

## 3. 如何定位判断规则

规则发现走**离线 research loop + 人工确认**流程：

1. **特征提取**：`scripts/extract_features.py` 对 19 篇训练样本做结构化特征对比，识别区分维度
2. **规则假设**：分析通过/不通过样本在各维度的差异，形成候选规则
3. **正则化**：能用正则捕获的格式信号（审批在场性、模板残留、占位符）写为确定性规则
4. **LLM 补充**：需要世界知识的语义信号（内容质量、领域自洽性）由 LLM 判断
5. **回测验证**：`eval/backtest.py` 全量回测，检查混淆矩阵和各类 Recall

**关键原则**：规则描述"判断维度"，不描述"训练集具体表面值"。不靠背答案。

## 4. 如何输出硬标签

judge 节点的裁决逻辑分三层：

### 第一层：确定性裁决（无 LLM）

```python
if all_passed:     → "通过"
if critical_fails: → "不通过"  # confidence ≥ 0.7 的失败
```

### 第二层：LLM 综合裁决

当存在低置信度失败（confidence < 0.7）且非全部通过时，调 LLM 综合权衡。prompt 内置裁决铁律（addendum §1.3），不在 Python 里写复杂加权算法。

### 第三层：Fallback

LLM 不可用时，默认不通过（安全侧）。

### 输出格式

对齐作业要求的 JSON：
```json
{
  "id": "sample_001",
  "dataset_type": "计划任务书",
  "intent": "判断创新程度是否足够",
  "label": "通过",
  "matched_rules": [{"rule_id": "R-03", "rule_name": "...", "evidence": "..."}],
  "reason": "简短判断说明"
}
```

## 5. 如何保证新文档上的泛化能力

这是本系统设计的**核心考点**，也是最容易出问题的地方。

### 5.1 问题诊断

LLM 天生倾向过拟合（作业 Hint #2）。第一版系统靠把训练集表面特征（KPI 具体数值 "85%/90%", 年份 "2026", 项目名 "输电线路覆冰预警..."）写死进正则，训练集准确率虚高至 94.7%，隐藏集必然断崖。

### 5.2 方法论：结构/语义二分 + 禁指纹

```
正则（判结构）              LLM（判语义）
─────────────────         ─────────────────
• 模板说明文字残留         • 摘要是否具体攻关路径
• 编号占位符成员           • KPI 与项目类型是否自洽
• 审批/签名「在场性」       • 预算是否合理匹配
• KPI 自我重复（模板复制）   • 创新点是否有实质方法论
─────────────────         ─────────────────
  通用，换文档也成立          需要世界知识，靠抽象维度判断
```

### 5.3 三条铁律（代码级约束）

1. **不匹配具体数值**：`\d{4}年\d{1,2}月\d{1,2}日` ✓（判"有日期"），`2026年\d{1,2}月\d{1,2}日` ✗（判"是2026年"）
2. **不给 LLM 看训练集样本**：prompt 和 few_shot 只用抽象准则或虚构示例，标注「示例仅说明判断逻辑，不是匹配模板」
3. **诚实基线**：指纹清零后训练集准确率 68.4%（纯规则）是真实起点，靠泛化能力涨分，不靠背答案

### 5.4 其他泛化保障

- **Intent 路由**：按意图激活规则子集，不全量暴露——陌生 intent 时兜底全量加载
- **少数类保护**：通过率仅 21%，judge 裁决铁律明确"倾向从严"，但不过度——100% 通过 Recall（4/4 通过样本不会全判不通过）
- **规则即代码**：YAML 文件可独立测试、版本控制、人工 review

## 技术栈

| 层 | 选型 | 说明 |
|----|------|------|
| 编排 | Python 顺序调用 | 3 节点线性流程，不需要状态机 |
| LLM SDK | Anthropic Messages API | 统一客户端，支持 Claude/DeepSeek 双后端 |
| 运行时模型 | claude-haiku-4-5 | 便宜，常规判断 |
| 难例模型 | claude-sonnet-4-6 | R-07 多维评分 |
| 文档解析 | doc-read CLI + python-docx | OfficeCLI 引擎，表格/编号完整 |
| Web | Gradio | 快速原型，满足评委 5 步流程 |
| 规则存储 | YAML | 可版本控制、人工 review |
| 数据契约 | Pydantic | Agent 间松耦合交接 |
