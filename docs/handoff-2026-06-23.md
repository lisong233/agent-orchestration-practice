## 0. 一句话定性

> **这是一道伪装成 ML 竞赛的 Agent 工程题。** 核心两件事：
> 1. **离线**用 LLM 从 20 个带标签样本里反推出审批规则，写成「按需加载的规则文件」
> 2. **在线**用极简 LangGraph 管线把「解析 → 按 intent 匹配规则 → 裁决」串起来，出网页

**不训练任何模型。不需要向量库。不需要多 provider 交叉验证。** 这些是 v1 的过度工程，已废弃。

题眼（作业原文第 7 行）：
> "利用模型的世界知识，在硬标签信号下发掘软标签（过程标签）的语义"

= LLM 本身就懂"什么样的电力科技项目算合格"，我们的活是**用少量样本把这种隐性知识校准成可执行、可迁移的规则**。

## 1. 涉及的 Agent 开发技能（作业第 14 行点名的 5 个）

| 技能 | 在本作业的落地方式 | 对应代码位置 |
|------|------------------|------------|
| **skill trigger** | 每类 intent → 路由到对应规则子集，**不把全部规则一次性塞进 prompt**（对应 hint①"不该一次性暴露"） | `app/router.py` |
| **spec-handoff** | Agent 间用结构化 JSON 文件交接：解析产出`DocFields` → 评估读它产出`RuleVerdict[]` → 汇总出`FinalLabel` | `app/schemas.py` 定义契约 |
| **compact** | 长文档（几千字表格）先压成结构化摘要再喂判断 Agent，省 token + 聚焦 | `app/agents/parse.py` |
| **长期记忆管理** | 规则库本身 = 长期记忆。离线沉淀进 `rules/*.yaml`，运行时只读 | `rules/` 目录 |
| **research loop** | **离线规则发现循环**：分析→提规则→回测→找错例→精炼。全作业最核心 | `scripts/discover_rules.py` |

## 2. 技术栈（最终确定版）

| 层 | 选型 | 决策理由 |
|----|------|---------|
| **编排** | LangGraph | 状态机 + 条件路由，3 节点足够。确实合适，不是过度工程 |
| **Web** | Gradio | `gr.Blocks` 快速搭上传/选类型/输 intent/看结果 |
| **推理 LLM** | **DeepSeek V4 Flash**（`deepseek-v4-flash`） | 主人指定。OpenAI 兼容、1M context、JSON 输出、thinking 双模、$0.14/$0.28 每 1M token，作业级别成本几乎为零 |
| **文档解析** | doc-read CLI（DOCX，底层已换 OfficeCLI `view html`→Markdown）+ LibreOffice（.doc→.docx） | 已有工具链，表格/编号一次全包 |
| **规则库** | YAML 文件 | 规则即代码，git 版本控制，人工可 review |
| **状态** | 内存 / SQLite（审计日志，可选） | 作业级别不需要 PostgreSQL |
| **部署** | NAS + Docker Compose | 主人指定。不强求公网，NAS 内网访问即可 |

### DeepSeek V4 Flash 接入（已查证 2026-06）

```python
# OpenAI SDK 直接用，只改 base_url + model
from openai import OpenAI
client = OpenAI(
    api_key=os.environ["DEEPSEEK_API_KEY"],
    base_url="https://api.deepseek.com",   # 注意：不要加 /v1（会变成 /v1/v1）

    model="deepseek-v4-flash",
    messages=[...],
    response_format={"type": "json_object"},  # 强制 JSON 输出

    # extra_body={"thinking": {"type": "enabled"}}, reasoning_effort="high"
```

关键事实：
- base_url = `https://api.deepseek.com`（OpenAI 格式）
- model = `deepseek-v4-flash`（`deepseek-chat` 别名 2026-07-24 弃用，别用）
- 支持 `response_format={"type":"json_object"}` → 解析/裁决 Agent 强制结构化输出
- 支持 thinking 双模式：默认非思考（快、便宜），裁决环节可开 thinking 提质
- 1M context → 长文档全塞得下，但仍建议 compact 省钱

## 3. 数据洞察（决定规则方向的关键）

### 标签分布
```
不通过: 16 (80%)   通过: 4 (20%)
```

### imbalance 的真实原因（对应 hint④）
**不是采样偏差，是业务事实**：大多数申报材料质量不达标。对比通过/不通过样本：

| 维度 | 通过样本（如"红外测温辅助定位装置"） | 不通过样本（如"负荷预测研究"） |
|------|-----------------------------------|------------------------------|
| 团队 | 真实姓名+职称+具体分工 | "成员10""成员11"占位符 |
| 创新点 | 3 个具体技术创新点，有方法论 | 套话堆砌 / 空泛 |
| 预算 | 材料费/测试费明细到小数点 | 笼统大数 / 空白 |
| 内容 | 技术原理分点详述 | 只有封面+承诺书（空模板） |

**→ 规则的核心信号 = "材料完整度 + 创新实质性 + 可量化性 + 团队真实性"**。这就是要反推的东西。

### 两种文档类型（结构完全不同，必须分流）
| 类型 | 识别特征 | 关键字段 |
|------|---------|---------|
| **计划任务书** | 标题"科技项目计划任务书" | 项目编号、KPI 表、考核指标、经费预算 |
| **立项申请书** | 标题"职工技术创新项目立项申请书" | 技术原理、创新点、人员表、审批签章 |

### ⚠️ 工程陷阱
- 10 个 `.doc` 老格式 → doc-read 读不了，**先批量转 .docx**：
  ```bash
  libreoffice --headless --convert-to docx "input.doc" --outdir ./训练集_docx/
  ```
- **doc-read 底层已换成 OfficeCLI**（2026-06-21 定型）：DOCX 主力引擎 = OfficeCLI `view html` → Markdown，
  编号/表格/图片一次全包（OOXML 解析外包给 OfficeCLI，~3s，缓存后 <1s）。对本作业是**利好**——
  这些文档里有大量表格（人员表/KPI 表/预算表），OfficeCLI 路线能完整保留表格结构，比纯文本提取强很多。
  - 解析 Agent 直接调 `doc-read <file>.docx` 即可，输出已是带表格的 Markdown
  - `--split-by heading` 可按章节切分，便于 compact 时定位"创新点""预算"等关键段落
  - 默认有缓存（文件级 mtime hash），P1 全量解析后重跑 <1s，调试友好

## 4. 系统架构

```
┌─ Gradio Web (NAS:7860) ──────────────────────────┐
│  ① 选类型(计划任务书/立项申请书)                    │

└──────────────────────┬───────────────────────────┘
                       │
┌──────────────────────▼───────────────────────────┐
│           LangGraph 管线（3 节点）                  │
│                                                   │
│  [parse] ──DocFields──> [match] ──Verdict[]──> [judge]
│   解析文档              按intent匹配规则           汇总裁决
│   →结构化字段           →逐规则打分+证据           →硬标签+依据
│      │                     │                        │
│      ▼                     ▼                        ▼
│  doc-read              router 选规则子集       DeepSeek(thinking)
│  +compact摘要          rules/<type>/*.yaml      置信度+reason
└───────────────────────────────────────────────────┘

离线（不在运行时）:
  scripts/discover_rules.py  ── research loop 反推规则 → rules/
```

**为什么 3 节点不是 4 节点**：作业明说"每多一个 agent 失败面翻倍"。v1 的"规则发现 Agent"挪到离线脚本，运行时不需要。

## 6. 核心代码骨架

### 6.1 数据契约 `app/schemas.py`（spec-handoff 的核心）

```python
from pydantic import BaseModel
from typing import Literal

class DocFields(BaseModel):
    """节点1 输出：解析+compact 后的结构化文档"""
    doc_type: Literal["计划任务书", "立项申请书"]
    title: str
    summary: str                 # compact 压缩后的核心内容
    fields: dict                  # 关键字段：团队/创新点/预算/KPI...
    raw_excerpt: str             # 保留原文片段供证据引用

class RuleVerdict(BaseModel):
    """节点2 输出：单条规则的评估结果"""
    rule_id: str
    rule_name: str
    passed: bool
    evidence: str                # 引用文档原文
    confidence: float

class FinalResult(BaseModel):
    """节点3 输出：最终交付格式（对齐作业 JSON 要求）"""
    id: str
    dataset_type: str
    intent: str
    label: Literal["通过", "不通过"]
    matched_rules: list[dict]
    reason: str
```

### 6.2 规则文件格式 `rules/计划任务书/R02-创新实质性.yaml`

```yaml
rule_id: R-02
rule_name: 技术路线具备实质创新
applies_to: 计划任务书
# skill trigger：哪些 intent 会激活这条规则
intent_keywords: [创新, 创新程度, 技术先进性, 实质突破]
description: |
  判断项目技术方案是否包含实质性创新，而非常规技术的简单堆砌。
criteria:
  pass:
    - 描述了具体的方法论创新（如算法改进、新融合方式）
    - 创新点可与现有技术明确区分
  fail:
    - 仅罗列常规技术名词
    - 创新点空泛、套话
few_shot:                        # 来自训练集的锚点示例（平衡正反例）
  - excerpt: "多源传感器融合的防外破识别技术：融合毫米波雷达与红外热成像..."
    verdict: pass
  - excerpt: "本项目针对...关键技术问题，拟开展核心技术攻关..."
    verdict: fail
```

### 6.3 LangGraph 编排 `app/graph.py`

```python
from langgraph.graph import StateGraph, END
from typing import TypedDict
from app.agents import parse, match, judge

class State(TypedDict):
    raw_doc: str
    doc_type: str
    intent: str
    fields: dict          # parse 产出
    verdicts: list        # match 产出
    result: dict          # judge 产出

def build_graph():
    g = StateGraph(State)
    g.add_node("parse", parse.run)
    g.add_node("match", match.run)
    g.add_node("judge", judge.run)
    g.set_entry_point("parse")
    g.add_edge("parse", "match")
    g.add_edge("match", "judge")
    g.add_edge("judge", END)
    return g.compile()
```

### 6.4 skill trigger 路由 `app/router.py`

```python
import yaml, glob

def select_rules(doc_type: str, intent: str) -> list[dict]:
    """按 intent 关键词匹配，只返回相关规则子集——不全量加载"""
    rules = []
    for path in glob.glob(f"rules/{doc_type}/*.yaml"):
        r = yaml.safe_load(open(path, encoding="utf-8"))
        if any(kw in intent for kw in r.get("intent_keywords", [])):
            rules.append(r)
    # 兜底：没匹配到就加载该类型全部（避免漏判）
    return rules or [yaml.safe_load(open(p, encoding="utf-8"))
                     for p in glob.glob(f"rules/{doc_type}/*.yaml")]
```

### 6.5 research loop `scripts/discover_rules.py`（最核心，半自动+人工确认）

```python
"""
规则发现循环（离线运行）：
  1. 读全部已解析文档 + 标签
  2. LLM 分析每个样本"为什么通过/不通过" → 提候选规则
  3. 用候选规则回测训练集 → 算 F1
  4. 找出预测错的样本 → LLM 分析 → 精炼规则
  5. 重复 3-4 直到收敛
  6. ★人工确认 checkpoint：导出规则草稿供人 review，确认后写入 rules/
"""
# 全自动跑循环，但第 6 步必须人工 review 才落地（主人要求保留人工通路）
```

## 8. 分阶段实施路径

| Phase | 任务 | 产出 | 验证点 |
|-------|------|------|--------|
| **P1 数据准备** | `.doc`→`.docx` 转换；全量结构化提取；标签提取 | `训练集_docx/`、`data/parsed/`、`data/labels.json` | 20 个文档都能解析出字段 |
| **P2 规则发现** ★ | 跑 `discover_rules.py` research loop；**人工 review** 规则草稿 | `rules/<类型>/*.yaml` | 训练集回测 F1 收敛、规则人工确认合理 |
| **P3 运行时管线** | 写 schemas/router/3 个 agent/graph | `app/` 可跑通单文档 | 输入1文档+intent 出正确 JSON |
| **P4 Web 界面** | Gradio Blocks 包壳 | `app/web.py` | 本地浏览器走通完整流程 |
| **P5 回测调优** | `eval/backtest.py` 全训练集回测；调规则/prompt | F1/Precision/Recall 报告 | 少数类（通过）召回率达标 |
| **P6 部署交付** | Docker 上 NAS；写 README.md + design.md | 公网/内网 URL + 文档 | 评委流程在 NAS 上可复现 |

**难度排序**（时间该花哪）：
```
🔴 P2 规则发现（决定成绩，最花脑力）
🟡 P3 编排 + P5 调优
🟢 P1 / P4 / P6（机械活）
```

## 9. design.md 要回答的 5 个问题（作业要求，提前备着）

1. **如何处理两类数据集** → doc_type 检测 + 分流到不同 rules 子目录
2. **智能体如何分工** → 3 节点 spec-handoff（parse/match/judge），数据契约见 schemas.py
3. **如何定位判断规则** → 离线 research loop 反推 + skill trigger 按 intent 激活
4. **如何输出硬标签** → judge 节点汇总 verdicts，加权/规则优先级裁决
5. **如何保证泛化** → 规则基于语义而非样本记忆；few-shot 平衡取例；intent 兜底全量加载
