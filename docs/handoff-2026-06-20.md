## 一、这是什么？

**「智能体编排业务判断挑战」**——AI 军团（AI Army）进阶开发课程的第二次作业。

### 一句话概括

设计一个多 Agent 协作系统，读取电力行业业务文档，根据评委给定的判断意图（如"创新程度是否足够"），自动发现并应用判断规则，输出 通过/不通过 + 命中规则 + 判断依据。

### 交付物

```
公网网页 URL
README.md
design.md
源代码仓库或 zip
```

### 评分维度

1. **方案与工程质量** — 设计是否合理，代码是否可维护
2. **硬标签准确率** — 在隐藏测试集上的 通过/不通过 正确率
3. **规则语义对齐** — 是否定位到了正确的判断规则

## 三、调研结论

完整调研报告：`d:\ClaudeWorkspace\agent-learning\AiArmy\.claude\memory\research\agent-orchestration-research.md`

### 3.1 核心架构推荐

```
Web 界面 (Gradio → 最终 FastAPI)
        ↓
LangGraph Orchestrator（状态机编排）
        ↓
┌───────────┬───────────┬───────────┬──────────┐
│ ①文档解析  │ ②规则匹配  │ ③综合判断  │ ④格式化   │
│  Agent    │  Agent    │  Agent    │  输出     │
└───────────┴───────────┴───────────┴──────────┘
```

### 3.2 四个 Agent 的分工

| # | Agent | 输入 | 输出 | 模型 |
|---|-------|------|------|------|
| ① | 文档解析 | 原始 DOCX/DOC | 结构化字段 JSON（项目名、创新点、预算、人员等） | DeepSeek-V3（便宜） |
| ② | 规则匹配 | 结构化文档 + intent + 规则库 | 每条规则的评估结果 + 证据引用 | Claude 4 Sonnet |
| ③ | 综合判断 | 所有规则评估结果 | 最终裁决 + 置信度 | Claude 4 Sonnet + GPT-4o 交叉验证 |
| ④ | 格式化 | 裁决结果 | 标准 JSON 输出 | 代码逻辑 |

### 3.3 规则发现管线（最核心）

从 MoRE-LLM 论文借鉴：

```
阶段 1: 初始规则提取
  LLM 逐条分析训练样本"为什么通过/不通过"
  → 提取候选规则 → 人工 review + 去重

阶段 2: 规则精炼（迭代循环）
  for each round:
    用当前规则集评估所有训练样本
    → 找出低置信度/预测错误的样本
    → LLM 分析 → 补充/修正规则 → 去重剪枝
  直到准确率收敛

阶段 3: 规则冻结 + 对抗验证
  构造边界测试样本 → 验证泛化 → 生成最终 SKILL.md 集合
```

### 3.4 关键挑战与对策

| 挑战 | 对策 |
|------|------|
| 隐藏测试集泛化 | RL-style 规则验证循环 >> 纯 SFT memorization |
| 数据极度不平衡 (4:16) | Class-few-shot + 少数类样本放 prompt 末端 |
| 规则不公开 | MoRE-LLM 迭代发现 + 规则即代码 |
| 不同 intent 触发不同规则 | Skill Trigger 模式：intent → 语义检索 → 激活规则子集 |
| 两种文档类型结构不同 | 类型检测 → LangGraph 条件路由到不同解析模板 |
| .doc 老格式 | LibreOffice headless 批量转 .docx |

### 3.5 技术栈推荐

| 层 | 推荐 | 说明 |
|----|------|------|
| 编排 | LangGraph v0.4 | 状态持久化、条件路由、HITL |
| Web | Gradio（原型）→ FastAPI（正式） | HuggingFace Spaces 免费部署 |
| 文档解析 | doc-read CLI（DOCX）+ LibreOffice（.doc 转换） | 已有工具链 |
| 主推理 LLM | Claude 4 Sonnet | 强推理能力 |
| 批量匹配 LLM | DeepSeek-V3 | 低成本高吞吐 |
| 评审 LLM | GPT-4o | 不同 provider 交叉验证 |
| 规则存储 | YAML 文件 + SKILL.md | 规则即代码，可版本控制 |
| 向量检索 | ChromaDB | 规则语义匹配 |
| 状态持久化 | SQLite（原型）→ PostgreSQL（生产） | LangGraph checkpoint |

## 六、关键决策记录

1. **编排框架选 LangGraph 而非 CrewAI/Dify**：需要精确控制执行流、条件路由、状态持久化。CrewAI 适合快速原型但复杂逻辑不可控。

2. **规则存储用 YAML + SKILL.md 而非硬编码**：规则即数据，可版本控制、热加载、独立测试。

3. **LLM 分层使用**：文档解析用便宜模型（DeepSeek-V3），规则推理用强模型（Claude 4 Sonnet），评审用不同 provider（GPT-4o）避免同源偏差。

4. **先 Gradio 后 FastAPI**：快速原型验证用 Gradio + HuggingFace Spaces 免费部署，正式提交前迁移到 FastAPI + 云服务器。

5. **Web 界面放在最后**：在规则发现和编排逻辑稳定之前，先专注核心管线。Gradio 可以快速包一层先用着。
