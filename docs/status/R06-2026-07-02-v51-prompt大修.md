## ② JSON mode 实测结论（2026-07-02 23:06）

| 测试 | 输入 | 输出 | 结论 |
|------|------|------|------|
| baseline | 诱导自然语言 | 📝 散文 | 符合预期 |
| `extra_body={"response_format":{"type":"json_object"}}` | 同一诱导 | 📝 仍为散文 | **参数被静默忽略** |

**结论：DeepSeek 的 `/anthropic` 兼容端点不支持原生 JSON mode。** 原生 JSON mode 仅在 OpenAI 端点 `api.deepseek.com` 可用。当前方案（prompt 末尾"请只输出JSON"+ `_extract_json` 三层正则兜底）实测零解析失败（NAS 6 次 + 19 次回测），维持现状。

**不改代码，仅记录为已知限制。** 若要硬保证，唯一路径是切 OpenAI SDK，但会推翻 S03「统一 Anthropic SDK」的决策。

## ③ prompt 维修详情

### 3.1 R-03/R-04 手写锚点（最高优先）

废掉字符串拼接，在 `match.py` 定义常量 `R03_DIMENSIONS` / `R04_DIMENSIONS`：

**R-03 技术方案实质性**：
```
维度1 — 技术方法具体性
  5分=有明确的技术路线，方法/原理/步骤具体到可复现，含个性化细节，换个项目名不再成立
  3分=有一定技术描述但较笼统，部分内容可套用于同类项目
  1分=仅有"针对XX关键问题拟开展攻关研制"这类空泛套话，换任何项目名都成立
维度2 — 创新点深度
  5分=创新点是具体可辨识的方法论/机理突破，说清"新在哪、比现有好在哪"
  3分=提出了创新方向但论述较浅，未说清与现有技术的实质区别
  1分=无创新点，或仅罗列"具有自主知识产权""填补空白"等口号式空话
```

**R-04 预算编制规范性**：
```
维度1 — 预算分项完整性
  5分=分项到多个具体科目，每项有明确金额，结构完整
  3分=有分项但科目偏少或部分金额缺失
  1分=仅有笼统总数、无分项明细，或预算表基本空白
维度2 — 金额合理性
  5分=金额与项目规模匹配，有单价×数量等计算依据，非拍脑袋整数
  3分=金额大体合理但偏整数/缺计算依据
  1分=金额明显不合理，或全是笼统大数无任何依据
```

### 3.2 堵死默认放水

`_eval_content_quality` 中 `s1_raw = dim1.get("score")` → 检查 `isinstance(s1_raw, (int, float))`，任一缺失即从严判不通过。不再默认 3 分。

### 3.3 KV cache 结构优化

所有 match 类 prompt 文档块前置：`{doc_block} → 项目名称+维度在后面`。同一篇文档 R-03/R-04/R-07 共享前缀。

### 3.4 统一 `wrap_for_llm()`

5 处 prompt 全接入 `sanitize.wrap_for_llm()`：
- `MATCH_USER`（通用单规则）
- `R07_MULTIDIM_USER`
- `CONTENT_QUALITY_USER`（R-03/R-04）
- `PARSE_USER`（parse.py）

handler 从 `.format(raw_excerpt=...)` 改为 `.format(doc_block=wrap_for_llm(...))`。

### 3.5 其他

- **judge.py**：删除 `JUDGE_SYSTEM`/`JUDGE_USER` 常量 + 未使用的 `chat_json` import
- **parse.py prompt 友好化**：强化"抽取不评判"导向，字段描述改为动作式
- **R-07 prompt**：追加 `{doc_block}` 占位符，文档块前置
- **design.md / README.md**：同步 `deepseek-v4-pro`→`deepseek-v4-flash`

### 3.6 max_tokens 上调

| 调用 | 旧值 | 新值 | 理由 |
|------|:--:|:--:|------|
| parse（`chat_json` 默认） | 2000 | 4000 | 最长输入，需提取多字段 |
| R-03/R-04 `_eval_content_quality` | 1200 | 3000 | 超大文档可能截断 evidence |
| R-07 `_eval_r07_multidim` | 1500 | 3000 | 三维输出 + evidence，对齐 |

## ⑪ 后续施工方向（供下一份 spec 参考）

以下三项是评审交付的"最后一步"——当前系统能跑、框架能判，但评委真正用起来还差这层皮。

### 11.1 真正的公网访问

**现状**：NAS 通过节点小宝穿透到 Tailscale 内网 `http://100.66.1.1:7860`（HTTP 200），仅 Tailscale 客户端可达。

**需求**：任何评委都能从公网打开，不需要装 Tailscale、不需要内网 IP。

**可选路径**：

| 路径 | 原理 | 复杂度 | 说明 |
|------|------|:--:|------|
| **A. frp/ngrok 等穿透** | 公网服务器做反向代理 → NAS:7860 | 低 | 最快能拿到公网 URL。需要一台有公网 IP 的机器（云服务器/VPS） |
| **B. Cloudflare Tunnel** | `cloudflared` tunnel → NAS:7860，自带免费 SSL | 中 | 需要域名绑定 Cloudflare，但免费且稳定 |
| **C. 直接公网暴露** | NAS 直连公网 IP + 端口映射 | 高 | UGOS 路由层不一定支持，风险大 |

**推荐先试 B**（免费 + 稳定），不行再 A（需要云服务器但可控）。

### 11.2 自定义 Key 输入

**现状**：DEEPSEEK_API_KEY 硬编码在 NAS 的 `.env` 里，评委用主人的密钥，主人掏钱。

**需求**：Web UI 加一个"API Key"输入框——评委可填自己的 key，系统用评委的 key 跑；留空则用主人的默认 key。

**改动点**：
- `web.py`：新增 `gr.Textbox(type="password", label="API Key（可选）")`
- `graph.py` / `llm.py`：`AuditPipeline.run()` 接受可选 `api_key` 参数，覆盖环境变量
- 一次会话内一个 key，刷新后清空（不落盘）

**风险**：评委填了错 key → LLM 调用全崩 → 系统按 `fallback` 从严全判不通过。需在 UI 提示"若填错 key 将无法使用 LLM 内容评分"。

### 11.3 用户识别 + 用量监测

**现状**：`logs/runs.jsonl` 记录每次判断（时间/文档/verdict/label），但无调用方身份、无 token 消耗统计、无法区分"谁的请求花了多少"。

**需求**：特别是评委用主人的默认 key 时，需要知道跑了多少篇、烧了多少 token、谁跑的。

**改动点**：

| 功能 | 实现 |
|------|------|
| **用户识别** | Web UI 加"评委姓名/编号"输入框（简单不验证）。写入 `runs.jsonl` 每条的 `user` 字段 |
| **Token 统计** | `llm.py` 的 `chat_json/chat_text` 从 Anthropic API 响应中取 `usage.input_tokens` / `usage.output_tokens`，写回 verdict/log |
| **用量面板** | Web 结果区下方加一个「用量」行：本次调用 token 数 + 累计调用次数 + 累计 token（从 `runs.jsonl` 汇总） |
| **主人密钥专用限额** | 若评委没用自定义 key（即走主人默认 key），在 `llm.py` 加一个简单的调用计数器 + 每日上限（如 200 次/天），超限返回"今日 API 额度已用尽，请使用自定义 key" |

**注意**：DeepSeek Anthropic 兼容端点的 `usage` 字段格式需实测确认（和原生 Anthropic API 可能有差异）。
