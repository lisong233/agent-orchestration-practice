## 一、项目目标

「AI 军团」— Agent 编排作业。构建多 Agent 系统，读取电力行业业务文档（计划任务书/立项申请书），根据评委输入的 intent 返回：硬标签（通过/不通过）+ 命中规则 + 判断依据。

**核心约束**：公开训练集仅用于理解任务，最终评测在隐藏数据集上进行。训练集准确率不直接计入成绩。LLM 天生倾向过拟合——规则需要泛化。

### P3 — 运行时管线 ✅

**架构**：3节点顺序管线（非 LangGraph StateGraph，线性流程不需要状态机）

```
[parse] 文档→结构化字段 → [match] 逐规则评估 → [judge] 汇总裁决→硬标签
```

**技术栈**：
- LLM：Anthropic Messages API（`claude-haiku-4-5` 运行时 / Sonnet 离线分析）
- 鉴权：复用 Claude Code 的 `ANTHROPIC_AUTH_TOKEN` 环境变量
- 依赖管理：`uv venv`（要求，非 pip venv）
- Web UI：Gradio（`app/web.py`，已写待测）

**代码结构**：
```
src/aiarmy/
├── schemas.py    ← Pydantic数据契约
├── llm.py        ← Anthropic SDK封装
├── graph.py      ← 管线编排（带过程输出）
├── web.py        ← Gradio界面
└── agents/
    ├── parse.py  ← 解析（正则快模式/LLM深模式）
    ├── match.py  ← 规则匹配（quick_check正则 + LLM fallback）
    └── judge.py  ← 裁决（critical失败→直接不通过 / LLM综合）
```

### P5 回测 — 完成

**结果**：

| 模式 | 综合 | 立项申请书 | 计划任务书 |

## 四、的关键决策与反馈

1. **训练集归拢**：`raw/`（源文件不可动）+ `可用/`（软连接入口）+ `convert_text/`（解析产物）。`训练集/` 整目录 gitignored。
2. **Infra 硬化**：`MINERU_VENV_PYTHON` 环境变量外置到 `settings.local.json` + `~/.bashrc`。Python 双入口决策规则写入 `platform.md`。
3. **代码放 `src/` 下**：`app/` → `src/aiarmy/`，不要散落根目录。
4. **用 uv 不用 venv**：`uv venv` + `uv pip install`，非 `python -m venv`。
5. **API 复用 Claude settings**：`ANTHROPIC_AUTH_TOKEN` 已有，不需要额外配 DeepSeek key。
6. **过拟合警告**：规则不能对着训练集写答案。引用 Hint #2 "LLM会天生倾向以过拟合的方式满足硬标签"。
7. **Docs 拆分**：`docs/spec/`（指导文件）+ `docs/status/`（工程现状），供下一个强模型接力。
8. **过程输出**：管线每个节点打印进度，但不能太多淹没上下文。

## 六、给下一个 Spec 的建议

### 核心挑战

**计划任务书的语义判断是唯一真正的难点**。立项申请书 100% 可以通过格式规则解决，不需要 LLM。但计划任务书中：
- 2篇「通过」和 8篇「不通过」共享同一个模板
- 区分它们的只有内容质量——这需要 LLM 的世界知识
- 当前的 Haiku-4.5 不足以完成这个任务

### 可能的改进方向

1. **升级判断模型**：R-07 用 Sonnet/Opus（贵但准），其他规则保持 Haiku（便宜）
2. **Few-shot 校准**：在 R-07 的 LLM prompt 中嵌入通过/不通过的正反示例
3. **多维度评分**：不只看「是否模板套话」，增加创新性/可行性/预算合理性的独立评分
4. **规则数量未知**：题目说"每类数据集约有 XX 条判断规则"，当前 7 条可能不够或方向不对
5. **Skill trigger 实现**：当前 router 只做了关键词匹配，需要真正的 intent 路由
