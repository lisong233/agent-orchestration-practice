## ① 节点小宝认知修正（前置）

### 1.1 异地组网 ≠ 内网穿透

节点小宝有两个独立功能，之前被混为一谈：

| 功能 | 机制 | IP/域名 | 协议 | 谁能访问 |
|------|------|---------|------|---------|
| **异地组网** | P2P 虚拟局域网 | `100.66.x.x` 组网 IP | 全部 TCP/UDP | 仅同账号已登录设备 |
| **内网穿透** | HTTP 反向代理 | `xxx.iepose.cn` 域名 | 仅 HTTP/HTTPS | 任何人（公开模式） |

之前的文档（包括 R06）将 `100.66.1.1` 标注为「Tailscale 内网」或「内网穿透」均为错误。`100.66.x.x` 是节点小宝异地组网分配的虚拟 IP，设备间 P2P 直连，不是 Tailscale，也不是内网穿透。

详见：`reference/jiepoint-xiaobao/README.md`（帮助中心 ~60 页完整文档镜像）

### 1.2 内网穿透实测：免费版不可用

节点小宝控制台提示「暂未购买穿透服务 · 尊享版会员赠送 1 条穿透权益」。免费用户无法创建内网穿透规则。

**与文档的出入**：帮助中心文档未明确标注内网穿透为付费功能，但控制台实际需要会员（30 元/月）。

## ② 公网访问方案对比

### 方案矩阵

| 方案 | 原理 | 零成本 | 零运维 | NAS 不停机 | 最终 |
|------|------|:--:|:--:|:--:|:--:|
| **Gradio share=True** | Gradio 官方隧道 → NAS:7860 | ✅ | ✅ | ✅ | ✅ **已采用** |
| 节点小宝内网穿透 | HTTP 反向代理 → 域名 | ❌ 需付费 | ✅ | ✅ | ❌ 放弃 |
| Cloudflare Tunnel | cloudflared → NAS:7860 | ✅ | ❌ 需域名 | ✅ | 备选 B |
| ngrok | ngrok 隧道 → NAS:7860 | ✅ 有限 | ✅ | ✅ | 备选 A |
| frp + VPS | VPS 公网 IP → frp → NAS | ❌ VPS 月费 | ❌ | ✅ | 太重 |
| HuggingFace Spaces | `gradio deploy` | ✅ | ✅ | ✅ | 长期方案 |

### 2.1 Gradio share=True（已落地）

**原理**：Gradio 启动时建立到 `gradio.live` 的 WebSocket 隧道，Gradio 官方服务器分配临时子域名，外部请求经该隧道转发到容器内的 7860 端口。

**改动**：
- `web.py`：`share=True` 参数
- `docker-compose.yaml`：`PYTHONUNBUFFERED=1` 环境变量（解决 Docker 日志不显示 share URL 的问题）

**NAS Docker 实测**：
```
* Running on public URL: https://fe41e1b5a3b9aa1bd0.gradio.live
HTTP 200 OK ✅
```

**限制**：
- 有效期约 1 周（尽力维护）
- 容器重启后 URL 会变
- 依赖 Gradio 官方隧道服务可用性

### 2.2 Cloudflare Tunnel（备选 B）

R06 推荐的方案。免费、稳定、自带 SSL。缺点是需要一个域名绑定 Cloudflare。

**未采用原因**：需要域名（额外的准备工作）。如果 Gradio 隧道不够稳定，这是优先备选。

### 2.3 ngrok（备选 A）

一行命令即可：`ngrok http 7860`。免费版有带宽/连接数限制。可用于快速应急。

### 2.4 HuggingFace Spaces（长期方案）

Gradio CLI 自带 `gradio deploy`，一键部署到 HF Spaces。免费、永久 URL、不需要自己的机器一直开机。缺点是 DeepSeek API key 需要作为 secret 配置，且 Spaces 的冷启动延迟可能影响体验。

## ④ Memory 文档组织建议

本轮触发了一个元问题：节点小宝、NAS 运维、公网访问、代理配置等网络相关文档分散在 memory/ 下不同位置，缺少统一的检索入口。

### 现状

| 文件 | 当前分类 | 主题覆盖 |
|------|---------|---------|
| `memory/nas-operations.md` | 运维手册 | SSH/异地组网/Docker/Mihomo |
| `memory/cloudcli-setup.md` | 外部服务 | CloudCLI/节点小宝组网/iPad |
| `memory/clash-verge-setup.md` | 外部服务 | 本地代理/Mihomo |
| `reference/jiepoint-xiaobao/README.md` | 外部资料 | 节点小宝全功能 |

### 建议：软分类（MEMORY.md 加「网络与远程」分组）

**不做**强文件夹拆分，因为：
- `nas-operations.md` 同时涉及网络（SSH/组网）和 Docker 服务编排 —— 放哪个文件夹？
- `cloudcli-setup.md` 同时涉及 CloudCLI 安装和节点小宝网络 —— 拆不开
- 强拆分破坏现有 git 历史和交叉引用

**做**：在 MEMORY.md 中新增「网络与远程访问」软分组，用注释标注文件归属，不移动物理位置。

示例结构（MEMORY.md 新增段落）：
```markdown
## 网络与远程访问

> 以下文件归属多个分类，此处提供网络视角的聚合索引。

- [[nas-operations]] — SSH 连接（节点小宝异地组网）
- [[cloudcli-setup]] — 远程 Web UI 访问
- [[clash-verge-setup]] — 代理配置
- jiepoint-xiaobao/README.md — 节点小宝完整文档
- R07-公网访问方案.md（本文件）— 公网暴露方案对比
```

### 额外建议：公共知识独立为 reference

节点小宝属于「外部产品的文档」，不是「我们的使用经验」。已按 reference 规范放在 `reference/jiepoint-xiaobao/`。如果后续有类似的公网访问通用知识（如 Cloudflare Tunnel 配置、frp 部署），建议同样走 `reference/` 而非 `memory/`。
