**critic 三项检查**（全确定性，零 LLM）：
1. evidence 非空、长度 ≥ 6 字
2. evidence 中 ≥8 字的中文片段能在 raw_excerpt 中找到（子串匹配）→ 证明引用了原文
3. 跳过系统 fallback 消息（"需LLM判断，安全侧从严"等）

**回环控制**：`_need_revision` 条件路由——`critic_ok or revision_count >= 1 → END`，否则回 match。封顶 1 轮回环。

**定向重评**：match 读到 critic_feedback 后，解析其中提到的 rule_id（正则 `R-\d{2}`），只重评点名规则，其余复用上轮 verdicts。

## ④ 系统架构最终态

### 5 节点 LangGraph 管线

```
[sanitize] → [parse] → [match] → [judge] → [critic] ⇄ [match]
  净化         解析      匹配      裁决       质量门    (封顶1轮)
                                                    ↓ END
```

| 节点 | 职责 | LLM？ | 输出 |
|------|------|:--:|------|
| sanitize | 净化注入，包裹数据边界 | ❌ | 干净 raw_text |
| parse | .docx→结构化字段 + 类型检测 | 🔵 可选 | DocFields |
| match | 逐规则评分（正则+LLM双轨） | 🔵 R-03/04/07 | RuleVerdict[] |
| judge | 内容优先裁决（tier分层） | ❌ | FinalResult |
| critic | evidence 质量门 | ❌ | critic_ok/feedback |

### 双层规则框架

```
元规则层（judge.py 确定性逻辑）
  M1 内容优先  M2 数据非指令  M3 必要非充分  M4 regime感知
                          ↓ 约束
语义规则库层（7 条 YAML + LLM prompt 维度）
  A类 advisory:    R-01 审批签章 → 永不裁决
  B类 conditional: R-02 承诺书 / R-05 模板 / R-06 占位符 → regime感知
  C类 content:     R-03 技术 / R-04 预算 / R-07 内容 → 主判据
```

### 文档解析三级降级

```
.docx/.doc 上传
  → ① doc-read CLI（OfficeCLI HTML→Markdown，开发环境）
  → ② LibreOffice headless（Docker 环境，表格/段落/编号完整）
  → ③ python-docx（纯文本兜底）
  → ④ 当纯文本读（终极兜底）
```

## ⑥ 诚实代价（必须读）

> ⚠️ 这一段是 S05 spec §⑥ 要求执行模型必须理解的护栏。

1. **立项申请书训练集准确率会下降**：因为 R-01 现在不参与裁决。训练标签中 2 篇通过 = 审批齐全（形式信号），我们不再把形式信号当主判据。这个下降是预期且正确的——S05 spec §⑥ 明确写了。

2. **无 LLM 时所有内容规则 fail**：R-03/R-04/R-07 在 quick_check 返回 None，无 LLM 时走 fallback 从严。这意味着纯规则模式下的综合准确率会低于 v4 的 89.5%。**不应用此数字作为成果指标。**

3. **真正的成果指标**（S05 spec §⑥ 定义）：
   - ① 三条路测试全绿 ✅
   - ② 反指纹稳健 ✅
   - ③ evidence 引用原文可复核 ✅（critic 质量门）
   - ④ 框架兑现「元规则+语义规则库」 ✅

4. **计划任务书训练标签仍是噪声**：8 篇不通过 + 2 篇通过在文本层面无法区分。内容优先框架在此类文档上的 label 准确率没有意义——重要的是 evidence 说服力。
