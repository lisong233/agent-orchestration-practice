## ⓪ 为什么需要这一轮

### 核心洞察（两次迭代）

**第一层**：训练集可能只是形式样本，**测试集可能全部格式齐全**（审批/签名/模板都完整），区分点纯粹在内容质量。→ R-03/R-04 的桩会导致立项申请书准确率崩。

**第二层（更深）**：测试集更可能是**形式审查完全空着，内容也是评委故意留白/模板填充**。评委的主观意愿就是让我们**辅助判断**——系统输出充分的理由和证据，评委审查我们的理由是否成立，再对比 ground truth 看我们判得对不对。

### 这意味着什么

**评分重心从「label 准确率」移到「理由说服力」**：

| 评分项 | 真正考点 |
|--------|---------|
| 硬标签评分 | 我们的 label vs ground truth |
| **规则语义对齐** | **matched_rules 里的 evidence 能不能说服评委？判断维度对不对？** ← 核心 |
| 方案与工程质量 | 架构合理性 |

**matched_rules 里的 evidence 和 reason，比 label 本身更重要。**

### 两条腿的真正价值

不是为了让 label 更准，而是**给评委展示完整的判断过程**：

```json
{
  "label": "不通过",
  "matched_rules": [
    {"rule_id": "R-01", "evidence": "审批意见缺失，日期为空白模板"},
    {"rule_id": "R-03", "evidence": "技术方案仅含模板套话，无具体技术方法和实验设计"}
  ],
  "reason": "形式审查：审批签章缺失。内容审查：技术方案缺乏实质性，仅含通用模板描述。综合判定不通过。"
}
```

评委看完判断：「形式确实空缺，内容确实空洞——理由充分，判断合理。」→ 得分。

`终审盲点扫描 agent` 的审计报告（`.claude/playground/final-audit.md`）发现了 4 个额外漏洞需一并修复。

**本轮目标**：两条腿（形式审查 + 内容审查）都做实。**每条 verdict 的 evidence 字段要从原文引用具体证据**，不只是"通过/不通过"的标签。
