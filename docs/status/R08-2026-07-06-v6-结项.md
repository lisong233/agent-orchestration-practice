## ① 多文档并发 + 翻页浏览

### 1.1 需求

题目 JSON 示例中 `"id": "sample_001"` 暗示多文档场景。原系统仅支持单文件上传，需升级。

### 1.2 实现

**上传**：`gr.File(file_count="multiple")` 原生支持。

**并发模型**：核心挑战是 `chat_json()` 为同步阻塞调用，在 async pipeline 中会阻塞 event loop。直接用 `asyncio.gather()` 无法真正并行。

方案：`loop.run_in_executor()` 把整条 pipeline 放进线程池。每条 pipeline 在独立线程跑自己的 `asyncio.run()`。上层 `asyncio.Semaphore(3)` 限制并发。

```
process_batch(files)
  └─ Semaphore(3)
       ├─ Thread-1: asyncio.run(AuditPipeline.run(doc1))
       ├─ Thread-2: asyncio.run(AuditPipeline.run(doc2))
       └─ Thread-3: asyncio.run(AuditPipeline.run(doc3))
```

`graph.py` 新增 `run_sync_quiet()` 函数，供线程池调用，静默模式不打印日志。

**翻页**：`gr.State` 存储所有结果 + `current_page` 索引。结果区域显示当前页文档的裁决 HTML，导航按钮切换 `current_page` 并重新渲染。首页（page=-1）仅显示输入表单，无结果空白区域。

### 1.3 验证

公网端到端测试：上传 3 份 docx → 并发处理 → "第 1/3 份" → 翻到 "第 2/3 份" → "第 3/3 份" → 返回首页。类型自动检测正常（🔍 计划任务书 / 立项申请书）。

## ② 浅色主题：?__theme=light 方案调研与实践

### 2.1 问题

主人厌倦了原暗色工业风主题。重写 CSS 为浅色系后，在主人 Edge/Chrome 上仍显示暗色背景——原因：Windows 系统深色模式触发 `prefers-color-scheme: dark`，Gradio 6 前端 `handle_darkmode()` 自动切换。

### 2.2 调研过程与教训

**第一次尝试**：CSS `color-scheme: light` + `@media (prefers-color-scheme: dark)` 覆写。无效——Gradio 的 Svelte wrapper 在 dark 下直接注入硬编码颜色 `rgb(31, 41, 55)`（`#1f2937`），CSS 变量覆写不够。

**第二次尝试**：更激进的 `[class*="svelte"]` 选择器 + `!important`。部分有效但不彻底。

**正确方案**：通过 Tavily + Context7 调研，发现 Gradio 前端源码中 `handle_darkmode()` 的优先级规则：**URL 参数 `__theme` 拥有最高优先级**。`?__theme=light` 命中后完全不执行暗色切换逻辑，比其他任何方案都可靠。

实现方式：`launch(head=...)` 注入 `<script>`，页面加载时检测 URL 是否已有 `__theme`，无则自动 `window.location.search` 重定向到 `?__theme=light`。一次 redirect 后该 session 内始终浅色。

### 2.3 教训

- **先查官方文档/源码，不要靠 CSS 硬覆盖**。Gradio 有内置的 `__theme` 机制，不需要我们自己发明。
- 调研过程见 `.claude/memory/research/gradio-light-theme-research.md`（Tavily + Context7 双源交叉验证结果）。

### 2.4 最终防御体系

三层防御：
1. `theme=gr.themes.Soft()` — 浅色基础主题
2. `head` 注入 `?__theme=light` 重定向 — **主力**，阻止 Gradio JS 暗色切换
3. CSS `@media (prefers-color-scheme: dark)` 覆写 — 兜底

## ③ 评委自定义 API Key

### 3.1 需求

评委会想用自己的 DeepSeek Key，也支持任意 Anthropic 兼容后端（如智谱、月之暗面）。

### 3.2 设计

核心难点：管线是多线程并发的，如何让每个线程用不同的 API Key 而不互相污染？

方案：`llm.py` 使用 `threading.local()` 存储覆写参数。`graph.py` 的 `run_sync_quiet()` 在线程入口处调用 `set_override()`，finally 中 `clear_override()`。`get_client()` 和 `chat_json()` 优先读取 `_override`。

```python
# llm.py
_override = threading.local()
_override.api_key = None
_override.base_url = None
_override.model = None
```

UI 三件套：
- API Key（密码框，留空使用服务端默认）
- Base URL（默认 `https://api.deepseek.com/anthropic`）
- 模型预设下拉：`deepseek-v4-flash` / `deepseek-v4-pro` / `自定义...`（自定义时显示文本框）

逻辑：填 API Key 则三件套必填；留空走 `.env` 默认。

### 3.3 适用范围

仅覆盖 DeepSeek 兼容端点（Anthropic SDK `/anthropic` 路径）。未设计多 provider 适配层——主人明确不需要。

## ⑤ 访问控制

### 5.1 需求

公网站点无任何认证，任何人可上传文档消耗主人的 DeepSeek 余额。

### 5.2 实现

模块级 IP 计数（内存 dict，重启清零）：

- 无密码：每 IP 限 1 次请求
- 正确密码（`aiarmy2026-review`）：无限制
- `process_batch()` 入口处调用 `_check_access()` 检查，处理完成后调用 `_record_usage()` 记录

密码字段放在 UI 顶级（不是折叠面板内），带有醒目的 `⚠️ 不输入密码：每 IP 仅限使用 1 次` 提示。密码值写在线下提交文件 `提交.txt` 中，不公开。

### 5.3 局限

IP 计数在容器重启后清零，且 NAT 环境下多个用户可能共享同一出口 IP。对评审场景够用，不适用于生产环境。
