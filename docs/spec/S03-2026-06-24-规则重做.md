## ⓪ 一句话背景

「AI 军团」作业：读电力业务文档（计划任务书/立项申请书）→ 输出硬标签（通过/不通过）+ 命中规则 + 依据。骨架已搭好（三 agent 分层 + 线性管线 + Pydantic 契约 + Gradio Web），**P1 数据 / P3 管线 / P5 回测都跑通了，训练集 78.9%**。

**但这个 78.9% 是假分。** 它靠把训练集的表面特征（KPI 具体数值、2026 年份、项目名词表）写死进正则换来的。隐藏评测集上这些指纹一个都不会出现，分数会断崖。**本轮任务就是把假分拆掉，换成能迁移的真本事。**

## ② 核心方法论转变：正则判「结构」，LLM 判「语义」

这是 v3 与 v1/v2 的根本分歧。**规则只描述「判断维度」，绝不描述「训练集的具体表面值」。**

### 2.1 留给正则的（真正通用的结构信号 — 这些换文档也成立）

| 信号 | 正则判什么 | 为什么通用 |
|------|-----------|-----------|
| 模板说明文字残留 | `填写说明`/`删除本提示`/`请在此处填写` 是否还在 | 任何认真填的文档都会删掉，这是格式硬伤 |
| 编号占位符成员 | `成员\d+`（成员1/成员2…未替换成真名） | 占位符没填=没认真做，与具体内容无关 |
| 审批/签名「在场性」 | 审批意见栏、负责人签名、日期栏**是否非空** | **判「有没有填」，不判「填的是哪年」** |
| KPI 各项数值是否雷同 | 抽出该文档内 KPI 列表，判**是否互相复制粘贴**（多个不同指标取值完全一致） | 判「自我重复」这个结构特征，不匹配任何具体数值 |

> ⚠️ 审批日期：改成判「日期栏是否被填写」（正则匹配 `\d{4}年\d{1,2}月\d{1,2}日` 这个**通用日期形态**，不锁定 2026），而不是判「是不是 2026 年」。

### 2.2 留给 LLM 的（需要世界知识的语义信号）

正则做不了的，全交 LLM（match.py 已有 LLM fallback 通道，prompt 用 addendum §1.2）。LLM 判断的是**抽象维度**，prompt 里**不准出现任何训练集的具体例子**：

- **项目名与 KPI 领域是否自洽**：纯软件/算法类项目却用「装置准确率/响应时间」这类硬件 KPI → 不匹配。（让 LLM 自己用电力领域知识判断「这是软件还是硬件项目」「这个 KPI 配不配」，不要给它写死的项目名清单）
- **摘要是具体攻关路径还是模板套话**：是否只有「针对 XX 关键技术问题，拟开展核心技术攻关」这类空话，没有具体技术方向。
- **预算与项目规模是否合理匹配**。

### 2.3 R-07 的 yaml 怎么改

`few_shot` 字段：**删除写死项目名的正反例**，改成「抽象的判断准则描述」。若一定要给示例，用**虚构的、不在训练集里的**例子，且明确标注「示例仅说明判断逻辑，不是匹配模板」。

## ③ 致命可用性 Bug：Web 只能吃 .txt，吃不下评委的 .docx

`src/aiarmy/web.py:22` 直接 `open(path).read()` 读上传文件。但题目明确「评委上传业务文档」= `.docx`（二进制 zip 包），文本读必然乱码 → 系统当场崩。

**修复**：在 `web.py` 的 `process_document` 里，上传文件若是 `.docx`，先经 doc-read 转纯文本再喂管线。

```python
# web.py 伪代码
import subprocess, tempfile
def to_text(path: str) -> str:
    if path.lower().endswith((".docx", ".doc", ".pdf")):
        # doc-read 是已装的全局 CLI，直接 subprocess 调
        out = subprocess.run(["doc-read", path], capture_output=True, text=True, encoding="utf-8")
        return out.stdout
    # .txt 直接读
    with open(path, encoding="utf-8", errors="replace") as f:
        return f.read()
```

> ⚠️ 部署到 Docker/NAS 时 doc-read 不一定在。两条路二选一，**默认走 A**：
> - **A（推荐）**：Docker 内装 LibreOffice + python-docx，写一个轻量 docx→text 函数兜底，不依赖 doc-read CLI。
> - **B**：把 doc-read 也打进镜像。重，不推荐。
> 本机开发阶段直接用 doc-read CLI 即可，docx 解析逻辑抽成 `src/aiarmy/io.py` 一个函数，方便替换后端。

**验收**：浏览器上传一个**真实 .docx**（不是预转的 txt），能跑出 JSON 结果，不报编码错。

## ④ LLM 客户端：统一一套 anthropic SDK（省钱 + 去冗余）

现状：`src/` 用 `anthropic` SDK（claude-haiku-4-5），`scripts/discover_rules.py` 用 `openai` SDK 调 DeepSeek。**这不是必须的分裂，是冗余。**

**事实**：DeepSeek 提供 Anthropic 兼容端点 `https://api.deepseek.com/anthropic`，用同一个 `anthropic` SDK 切 `base_url` + `model` 就能调。

**改法**：`src/aiarmy/llm.py` 的 `get_client()` 支持按用途切后端，全程只用 `anthropic` SDK：

```python
# 运行时判断量大 → 走便宜的 DeepSeek；离线规则发现/难例 → 走强的 Claude
def get_client(backend: str = "claude") -> Anthropic:
    if backend == "deepseek":
        key = os.environ["DEEPSEEK_API_KEY"]
        return Anthropic(api_key=key, base_url="https://api.deepseek.com/anthropic")
    key = os.environ.get("ANTHROPIC_API_KEY") or os.environ.get("ANTHROPIC_AUTH_TOKEN")
    return Anthropic(api_key=key)
```

`discover_rules.py` 删掉 `from openai import OpenAI`，改用 `llm.chat_json(..., backend="deepseek")`。`requirements.txt` 去掉 `openai`。

> 模型分工：运行时 match/judge 的 LLM 调用用便宜模型（DeepSeek flash 或 claude-haiku）；离线规则发现/难例分析用强模型（Sonnet/Opus）。**别在运行时管线里调贵模型，量大烧钱。**

## ⑤ 计划任务书的真难点（这是分数的胜负手）

立项申请书 9/9 靠结构规则就能解决（审批/承诺书在场性 + 模板残留 + 占位符），**不要在它身上花 LLM**。

真正的难点全在计划任务书：2 篇「通过」和 8 篇「不通过」**共用同一模板**，结构层面几乎无差别，区分它们**只能靠内容质量的语义判断**。状态报告里 3/5/6/8 这 4 篇格式全合规，是 Haiku 判不准的硬骨头。

**改进方向（按性价比排序，至少做前两条）**：
1. **难例升级强模型**：R-07 的 LLM 路径换 Sonnet（贵但准），其余规则保持便宜模型。
2. **多维独立评分**：别只问「是不是套话」一个笼统问题。拆成三个独立打分（技术具体性 / KPI 领域自洽性 / 预算合理性），每维 1-5 分 + 依据，再综合。维度分开问，LLM 判得更稳。
3. **（可选）校准而非记忆**：若要给 few-shot，给的是「判断准则」不是「训练集样本」。

> ⚠️ 别为了把训练集 4 篇难例刷对而回头加指纹。**宁可这 4 篇错，也不能过拟合**——隐藏集才算分。诚实基线 78.9% 是可接受的起点，靠泛化能力涨分，不靠背答案。
