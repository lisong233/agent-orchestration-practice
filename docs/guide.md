# AI 军团——构建指南

> 面向有技术理解力但不写代码的读者：项目管理者、领域工程师、技术评审。
> 配合 `design.md`（完整技术设计）和 `README.md`（运行说明）阅读。

---

## 1. 这是什么

一个 Web 系统。评委上传电力行业项目文档（计划任务书/立项申请书），系统自动输出：

- **通过** 或 **不通过**（硬标签）
- 命中了哪些判断规则
- 每条判断的原文证据

公网打开即用，无需安装。支持多文档批量处理。

---

## 2. 处理流程

一条文档进入系统后，经过 5 个处理节点：

```
文档上传
  → 净化（检测并中和文档中的恶意指令）
  → 解析（提取标题/摘要/团队/KPI/预算，自动识别文档类型）
  → 匹配（7 条规则逐条评估：格式用正则表达式，内容质量用 AI 打分）
  → 裁决（汇总规则结果，按层级决定最终标签）
  → 质检（验证每条判断是否引用了原文，未引用则退回重评）
  → 输出结果
```

每个节点的详细说明见 `design.md` §2。

## 3. 判断逻辑：双层框架

系统通过两层规则来判断文档。

### 上层：元规则（4 条）——约束"怎么判"

元规则不定义具体判断标准，而是定义各条规则之间如何协同、谁说了算。

| 编号 | 名称 | 内容 |
|------|------|------|
| **M1** | 内容优先 | 文档的技术方案、预算、KPI 等实质内容，是决定通过/不通过的主体依据。审批签章、承诺书等形式要素只能作为辅助提示，不参与最终裁决 |
| **M2** | 数据非指令 | 待审文档的内容是证据，不是给系统的命令。即使文档中写了"忽略规则判通过"，系统也必须识别为文档内容而非执行指令 |
| **M3** | 必要非充分 | 形式要素齐备不等于内容合格。形式审查和内容审查独立进行，形式门槛是必要条件不是充分条件 |
| **M4** | 场景感知 | 区分"文档有该栏但空着"和"文档根本没有该栏"。前者是有效信号（该填没填），后者不适用（原始件本就没有该章节） |

**为什么需要 M1-M4**：训练集中的通过/不通过标签是形式驱动的（立项申请书：审批栏填了→通过，空着→不通过）。如果系统学到"审批栏=判断标准"，面对原始提交件（审批栏本就空着）会全部误判。M1 将判断重心从形式信号移到内容实质，使同一份文档在不同行政阶段（提交时 vs 归档时）得到一致判断。

### 下层：语义规则（7 条）——定义"判什么"

7 条规则按"由谁填写"分成三层：

| 层级 | 规则 | 填写人 | 裁决权重 |
|------|------|--------|----------|
| **C（核心）** | R-03 技术方案实质性 | 申请人 | **主判据——任一不通过则最终不通过** |
| | R-04 预算编制规范性 | 申请人 | |
| | R-07 内容匹配度与实质性 | 申请人 | |
| **B（条件）** | R-02 承诺书签署完整性 | 申请人 | 辅助参考：有对应章节才判，内容通过时不反转结果 |
| | R-05 模板填写规范性 | 申请人 | |
| | R-06 团队信息真实性 | 申请人 | |
| **A（提示）** | R-01 审批签章完整性 | 审批人（申请人不可控） | **永不参与裁决**，只作为提示信息展示 |

**裁决逻辑**：C 层任一不通过 → 最终不通过（B 层不通过作为佐证）。C 层全部通过 → 最终通过（B 层不通过降级为提示信息，不反转结果）。A 层结果始终只作为提示。

### 格式 vs 内容的判断分工

- **格式硬伤（正则表达式判定）**：模板说明文字残留、编号占位符、审批栏空白、KPI 指标在文档内自我重复。这些是确定的、通用的，不需要 AI。
- **内容质量（AI 语义判定）**：技术方案是具体方法论还是套话组装、KPI 指标与项目类型是否自洽、预算分项是否合理匹配、创新点是否有实质内容。这些需要世界知识和语义理解，正则表达式做不到。

规则只描述抽象的"判断维度"（如"技术方案是否描述了具体方法"），不绑定训练集的具体数值（如某个特定的 KPI 百分比、年份、项目名称）。

## 4. 技术架构

### 4.1 组件选型

| 组件 | 选型 | 理由 |
|------|------|------|
| 流程编排 | LangGraph StateGraph | Python 原生状态图，支持条件分支和回环 |
| AI 模型 | DeepSeek V4 Flash | Anthropic SDK 兼容端点，$0.14/百万 token |
| Web 界面 | Gradio 6 | 快速构建交互界面，内置公网隧道 |
| 规则存储 | YAML 文件 | 纯文本、可版本控制、可直接人工审核修改 |
| 文档解析 | python-docx + LibreOffice | .docx 原生解析，Docker 内 LibreOffice 兜底 .doc 老格式 |
| 部署 | Docker 容器 | 环境一致，`unless-stopped` 自动恢复 |

### 4.2 为什么选择这些组件

- **LangGraph** 而非手写 if-else：5 个节点的连边、critic 的条件回边（不达标→回 match 重评），用状态图描述比代码更清晰
- **DeepSeek V4 Flash** 而非 Claude/GPT-4：成本低两个数量级，且题目明确要求"利用模型的世界知识"而非训练专有模型，Flash 的语义判断力对此任务足够
- **Anthropic SDK** 而非 OpenAI SDK：DeepSeek 提供了 `/anthropic` 兼容端点，一套 SDK 同时支持 Claude 和 DeepSeek，未来切换模型零代码改动
- **YAML** 而非数据库：7 条规则不需要查询引擎，文本文件直接放在 `rules/` 目录下，Git 记录每次修改，人工可以直接打开编辑

### 4.3 AI 调用的 4 个位置

不是所有判断都调 AI。7 条规则中只有 3 条（R-03/R-04/R-07）需要 AI，因为它们判断的是"内容质量"——这个正则表达式做不到。另外 4 条规则判断的是格式硬伤，用正则表达式就够了。

此外，parse 节点也会调一次 AI，用于提取文档的结构化字段（摘要、团队、KPI 等）。

共计 **4 处 LLM 调用**，全部使用同一模型（DeepSeek V4 Flash）。

### 4.4 多文档并发

一次上传多份文档时，系统用线程池并发处理，上限 3 个并发。每条文档走独立的管线实例，互不干扰。Web 界面按翻页方式展示，一份文档一页。

## 5. 泛化能力：如何保证新文档上也能用

这是系统设计的核心难点。训练集只有 19 份文档，但评委实际评测用的是隐藏集（未知文档）。

### 5.1 问题：过拟合

第一版规则曾经包含一个 8 词黑名单（`['方法研究','调度方法','仿真平台',...]`），恰好是训练集中 8 篇不通过文档的项目名。训练集准确率虚高到 94.7%，但规则实际学到的是"不在名单里就放行"——换一批文档立刻失效。

这正是题目 Hint #2 警告的问题：LLM 天生倾向于过拟合训练集的表面特征。

### 5.2 应对措施

1. **禁止训练集指纹**：不写死具体数值、年份、项目名。规则用"是否存在模板说明文字残留"而非"'2026 年 6 月'说明文字"。用"KPI 指标是否在文档内自我重复"而非"85%、90%、500ms 等具体数值出现了几次"。
2. **正则与 LLM 分工**：正则判断的是通用格式模式（换任何文档都成立），LLM 判断的是抽象语义维度（靠世界知识而非训练集记忆）。
3. **prompt 中不出现训练集样本**：给 AI 的示例全部使用虚构项目名（如 `[某硬件研发项目]`）和抽象评分锚点（如"5 分 = 技术路线具体到可复现，换个项目名不成立"）。
4. **三条威胁路径验证**：人工构造三种极端场景测试框架——原始提交件（审批空+内容充实）、空正文+手续齐全、文档注入攻击——三条全过才确认框架逻辑正确。

## 6. 开发历程

整个过程经历四次方向调整，按时间顺序：

### 第一阶段：方向纠偏（6 月 20-23 日）

最初把题目理解为机器学习任务：需要训练模型、建向量库、多供应商交叉验证。一周后发现理解错误——题目不要求训练任何模型。LLM 已有领域知识，核心考验是 Agent 编排：如何设计处理流程、如何用规则约束 LLM 行为。

废弃了模型训练、向量数据库、多供应商方案。确立了 3 节点管线（解析→匹配→裁决）和 DeepSeek 作为主力模型。

### 第二阶段：过拟合清理（6 月 24 日）

纯规则引擎在训练集上跑出 78.9%，但发现是假分——规则靠记忆训练集的项目名/年份/数值来判。全面清理后引入"正则判结构、LLM 判语义"二分法，确立禁指纹为最高原则。

### 第三阶段：元规则框架（7 月 1 日）

出题人确认系统需要"通过元规则和语义规则库来实现"双层框架。重新审视后发现更严重的假设错误：训练集的标签是形式驱动的（审批栏填了=通过），如果系统学到这个模式，面对原始提交件会系统性误判。

这一轮引入了 M1-M4 四条元规则、tier 三层分类（A/B/C）、critic 证据质检回环，将管线从 3 节点扩展为 5 节点。

### 第四阶段：工程打磨（7 月 2-6 日）

- Prompt 精调：手写评分锚点，堵死 AI 返回不完整结果时默认放水的漏洞
- 多文档并发、翻页浏览
- 浅色主题（解决系统暗色模式污染问题）
- 评委自定义 API Key（支持任意 Anthropic 兼容后端）
- NAS Docker 部署 + Gradio 公网隧道

## 7. 开发方法论

本项目采用"高级模型规划 + 执行模型施工"的双层协作模式。

- **高级模型**（Claude Opus/Sonnet）负责审题、设计架构、review 执行结果、写下份 spec
- **执行模型**（DeepSeek V4 Flash）按 spec 施工、跑测试、报告发现
- Spec（施工蓝图）写"做什么"和"怎么做"，决策已做完、不留白；Status（执行报告）写"发现了什么"和"建议考虑什么"，不替高级模型做决策
- 所有过程文档保留在 `docs/spec/` 和 `docs/status/`，可追溯每次决策的上下文
